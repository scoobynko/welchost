## 1. What Welchost is

**Welchost** (a pun on **welc**ome + g**host**) is a macOS-only CLI tool that
creates and manages a welcome banner for the [Ghostty](https://ghostty.org)
terminal. When you open a Ghostty window, you get a big ASCII-art greeting with
optional color/gradient, a border, and system info.

### The Ghostty chain (how the banner actually shows up)

```
Ghostty launches
  ‚Üí ~/.zshrc sources ~/.config/ghostty/welcome.zsh   (guarded by a sentinel block)
    ‚Üí welcome.zsh runs: python3 ~/.config/ghostty/welcome_banner.py
      ‚Üí welcome_banner.py renders the banner to the terminal
```

Three links in the chain, three files Welchost owns. Welchost edits `~/.zshrc`
only to add/remove one guarded `source` line. Everything else lives in
`~/.config/ghostty/`.

### Hard rule: never touch Ghostty's own config

Welchost manages `welchost.toml`, `welcome.zsh`, and `welcome_banner.py`. It
**never** reads, writes, or deletes `~/.config/ghostty/config` or
`config.ghostty` ‚Äî those belong to Ghostty.

### Ghostty / interactive gate

The `.zshrc` line is always gated so the banner only runs in an interactive
Ghostty shell:

```zsh
[[ "$TERM_PROGRAM" == "ghostty" && $- == *i* && -r ~/.config/ghostty/welcome.zsh ]] && source ~/.config/ghostty/welcome.zsh
```

### Role of each module

- **detect.py** ‚Äî Resolves *where* files go (`get_config_dir()`, `get_zshrc()`)
  and *whether* Ghostty is present (`ghostty_installed()`). Holds the module-level
  `DEV_MODE` flag. This is the lowest layer ‚Äî depends on nothing else in the package.
- **config.py** ‚Äî The `WelchostConfig` dataclass mirrors `welchost.toml` exactly.
  `load_config()` reads it (returns `None` if absent), `save_config()` writes it.
  Uses `detect.get_config_dir()` for paths. Depends only on detect.
- **themes.py** ‚Äî The 12 named themes as plain data, each convertible to a
  `WelchostConfig`. No I/O.
- **generator.py** ‚Äî Renders `welcome.zsh` and `welcome_banner.py` from a config
  via Jinja2, and manages the `~/.zshrc` sentinel block (inject/remove, with
  backup). Uses detect for paths and config for the data. The `reset` logic lives
  here too (remove files + sentinel).
- **cli.py** ‚Äî The Typer app. Subcommands `config`, `preview`, `reset`, `doctor`,
  `version`. Root callback handles the global `--dev` flag. Importing the TUI is
  deferred (lazy import inside the command) so non-TUI commands stay fast and the
  core has no dependency on Textual.
- **update.py** ‚Äî Self-update support (core, no Textual). Checks PyPI for a newer
  release (`latest_version`, fails soft to `None` when offline), compares versions,
  detects the install method (brew/pipx/pip), and resolves/runs the upgrade command.
  On launch the TUI checks in a background thread and, if newer, prompts to update.
- **telemetry.py** ‚Äî Anonymous, opt-out usage analytics (core, no Textual).
  `track_launch()` mints/persists a random UUID in `analytics.json` (a sidecar ‚Äî
  **never** in `welchost.toml`, so the config round-trip stays lossless) and POSTs
  an `install`/`launch` event to PostHog. **Opt-in** (GDPR/ePrivacy-safe for EU):
  nothing is sent until the user consents. `is_configured()` = key + non-dev +
  no opt-out env; `needs_consent_prompt()` is true when configured but the choice
  is `unset`; the TUI then shows a consent modal and calls `record_consent()`
  (which persists the decision in `analytics.json` and, on grant, sends the first
  `install`+`launch`). `track_launch()` sends a `launch` only once consent is
  granted. Fails soft like `update.py`. The PostHog key is **never committed** ‚Äî
  it resolves from `WELCHOST_POSTHOG_KEY` env, else a gitignored `_secrets.py`
  that CI bakes into the build at release time (hatchling `artifacts` in the
  global `[tool.hatch.build]` table ‚Üí both sdist AND wheel, since Homebrew
  installs from the sdist). A plain checkout has no key ‚Üí no-op. Also disabled in
  DEV mode (unless `WELCHOST_TELEMETRY_FORCE=1`, a hatch for testing the prompt)
  or via `WELCHOST_NO_TELEMETRY` / `DO_NOT_TRACK`. `reset` removes
  `analytics.json`. Both phone-home paths (this + `update.py`) use
  `net.ssl_context()` (certifi-backed) so they work on Pythons without a system
  CA bundle.
- **tui/** ‚Äî Textual UI. **Depends on core, never the reverse.** Core modules must
  never `import` from `welchost.tui`.

### Dependency direction (strict)

```
tui ‚Üí core (detect, config, themes, generator)
```

Never import tui from core. Never hardcode paths ‚Äî always go through
`detect.get_config_dir()` / `detect.get_zshrc()`.

## 3. Stack decisions (and why)

| Concern | Choice | Why |
|---|---|---|
| CLI framework | **Typer** | Type-hint based, free `--help`, shell completion, clean callback for `--dev`. |
| TUI | **Textual** | Full-screen reactive widgets; live preview re-render via reactive attrs. Not `blessed`/`urwid` because reactivity + CSS-like styling make the live-preview wizard tractable. |
| ASCII art | **pyfiglet** | 428 fonts bundled as a pip dependency ‚Äî no system `figlet` binary to install, so `brew install welchost` is self-contained. **Never shell out to `figlet`.** |
| Rendering | **Rich** | Colors, panels, per-character gradient via `Color.blend()`. Textual is built on Rich, so the same render code works in TUI preview and in the generated `welcome_banner.py`. |
| Config format | **TOML** via stdlib `tomllib` (read) + `tomli_w` (write) | `tomllib` is stdlib on 3.11+; `tomli_w` for writing. Human-editable, typed. |
| Templating | **Jinja2** | Generate `welcome.zsh` and `welcome_banner.py` from config deterministically. |
| Tests | **pytest** + `typer.testing.CliRunner` | Standard; CliRunner exercises the real CLI. |
| Min Python | **3.11** | `tomllib` landed in 3.11. |
| Platform | **macOS only** (for now) | Ghostty + zsh assumptions. |

## 4. Config schema ‚Äî `welchost.toml`

Stored at `~/.config/ghostty/welchost.toml` (or `dev-home/.config/ghostty/` in
DEV mode). It is the **single source of truth**. `welcome.zsh` and
`welcome_banner.py` are always regenerated from it and must never be hand-edited.

```toml
[banner]
text = "Welcome"
font = "slant"
align = "left"         # left | center | right  (position on screen)
color_mode = "solid"   # solid | gradient

[color.solid]
value = "cyan"         # Rich color name OR hex "#rrggbb"

[color.gradient]
start = "cyan"
end = "magenta"
direction = "horizontal"   # horizontal | vertical | diagonal

[decoration]
border_style = "none"    # panel | box | double | rounded | ascii | none  (default: none)
border_color = "magenta"

[ornament]
name = "none"            # none | ghosts | stars | bars | dots  (flanks the banner)

[info]
show_user = true
show_datetime = true
show_host = false
show_os = false
show_uptime = false
show_shell = false
show_python = false
show_ip = false

[meta]
welchost_version = "1.0.0"
created_at = ""          # ISO8601 timestamp set on first save
```

### Valid values

- `banner.align`: `left | center | right` (position of the banner across the terminal width)
- `banner.color_mode`: `solid | gradient`
- `color.gradient.direction`: `horizontal | vertical | diagonal`
- `decoration.border_style`: `panel | box | double | rounded | ascii | none`
- `ornament.name`: `none | ghosts | stars | bars | dots` (predefined ASCII blocks
  flanking the banner; see `welchost/ornaments.py`)
- Colors: any Rich color name (`cyan`, `bright_green`, `magenta`, ‚Ä¶) or `#rrggbb`.
- `info.*`: all eight flags exist in the schema, but only `show_user` and
  `show_datetime` default on and are exposed in the wizard; the rest are for TOML
  power users.

`WelchostConfig` is a dataclass that mirrors this structure (nested dataclasses
or flat fields ‚Äî implementer's choice, but round-trip through TOML must be
lossless). Provide a `DEFAULTS`/`WelchostConfig.default()` factory matching the
above.

## 5. CLI interface

```
welchost              ‚Üí full interactive TUI (alias of `welchost config`)
welchost config       ‚Üí TUI: create wizard if no config, edit menu if config exists
welchost preview      ‚Üí render current banner to stdout (no TUI, no file changes)
welchost reset        ‚Üí remove all welchost files + .zshrc injection (confirms first)
welchost version      ‚Üí print version

welchost doctor       ‚Üí [DEV ONLY] Ghostty/env/chain diagnostics. Hidden from
                        `--help` and refuses to run outside dev mode.
```

Global flag (root callback): `--dev` sets `detect.DEV_MODE = True` **before** any
subcommand runs. `WELCHOST_DEV=1` env var is equivalent. The TUI entry point is a
lazy import; if Textual isn't importable for some reason the core commands must
still work.

`preview` reads the existing `welchost.toml`; if none exists it renders a
sensible default and says so. `doctor` is a **development-only diagnostic**:
hidden from the user-facing CLI *and* the TUI menus, and it refuses to run
outside dev mode (`--dev` / `WELCHOST_DEV=1`), exiting non-zero with a message.
In dev it reports each chain link (Ghostty, env, files, sentinel) as a finding,
not a crash. All user-facing output goes through a Rich `Console` ‚Äî **never bare
`print()`**.

Every screen has a `BannerPreview` widget (uses a `render()` override, not
`update()` ‚Äî Textual 8.x measures auto-height widgets via the render path) that
re-renders on any input change. It renders the same way `welcome_banner.py` will,
so what you see is what you get.

- No emoji anywhere ‚Äî pixel/block glyphs only. First screen shows the `Logo`
  splash: a pixel-art ghost beside a `double_blocky` "welchost" wordmark, a
  `~/welchost ‚ùØ` prompt, and the tagline. Menu items use accent block glyphs
  (`‚ñ¶ ‚óß ‚óà ‚ñ© ‚Ä¶`) as bullets.
- Menus: `ListView`, focused on mount, `‚Üë/‚Üì` + `enter`.
- Template picker (`TemplateList`): a single `ListView` of templates with a live
  `BannerPreview` below ‚Äî `‚Üë/‚Üì` browses (preview updates live), `enter` opens the
  wizard at step 1 (so text/font stay editable), `esc` back. A vertical list,
  not a grid, so arrow nav is unambiguous.
- Wizard: step content is Tab-navigable; **`ctrl+‚Üê` / `ctrl+‚Üí`** change steps
  (plain arrows are left to text inputs), `ctrl+s` saves & installs from
  anywhere. Each step auto-focuses its first control. Color presets and mode are

## 8. Gradient rendering

Gradient is per-character RGB interpolation between `gradient.start` and
`gradient.end`. Each character's blend factor `f ‚àà [0, 1]` comes from its
position in the **whole art block**, per `gradient.direction`:

- `horizontal` ‚Äî `f = col / (maxwidth - 1)` (left‚Üíright across each line; the default).
- `vertical` ‚Äî `f = row / (rows - 1)` (top‚Üíbottom; each line is one solid blend).
- `diagonal` ‚Äî `f = (col/(maxwidth-1) + row/(rows-1)) / 2` (top-left‚Üíbottom-right).

```python
def factor(col, row, dx, dy, direction):  # dx = maxwidth-1, dy = rows-1
    if direction == "vertical":
        return row / dy
    if direction == "diagonal":
        return (col / dx + row / dy) / 2
    return col / dx  # horizontal
```

This identical logic lives in both the TUI/CLI preview (`render.py`,
`render_art` ‚Üí `_gradient_factor`) and the generated `welcome_banner.py`
(`colorize_line` ‚Üí `_factor`), so the preview is WYSIWYG. The dimensions
(`maxwidth`, `rows`) are taken from the art block only ‚Äî info lines stay plain.
Solid mode just applies one style to the whole block.

## 9. `.zshrc` sentinel injection

Always use sentinel markers, never double-inject, always back up first.

```zsh
# >>> welchost >>>
[[ "$TERM_PROGRAM" == "ghostty" && $- == *i* && -r ~/.config/ghostty/welcome.zsh ]] && source ~/.config/ghostty/welcome.zsh
# <<< welchost <<<
```

Rules:
- **Backup** before any edit: `~/.zshrc.welchost.bak.<timestamp>`.
- **Idempotent**: if the sentinel block already exists, replace its contents
  rather than appending a second block. Detect by the marker lines.
- **Never blindly append** ‚Äî always scan for the start marker first.
- **Reset** removes everything between the markers (inclusive), leaving the rest
  of `~/.zshrc` untouched.
- If `~/.zshrc` doesn't exist, create it.
- In DEV mode the target is `dev-home/.zshrc` (via `detect.get_zshrc()`).

## 10. Generated files

### `welcome.zsh` (thin shim)

Just runs the renderer; no logic. Roughly:

```zsh
#!/usr/bin/env zsh
# Generated by welchost ‚Äî do not edit. Edit welchost.toml and regenerate.
python3 "${HOME}/.config/ghostty/welcome_banner.py"
```

### `welcome_banner.py` (the renderer)

A standalone Python script (depends only on Rich, which the user has via the
installed welchost venv ‚Äî or guard the import) that:
- Builds the pyfiglet text from `banner.text` / `banner.font` (native size).
- Applies solid color or per-character gradient (the `render_gradient` pattern).
- Wraps in the chosen border (`panel`/`box`/`double`/`rounded`/`ascii`/`none`)
  via a Rich `Panel` (or plain for `none`).
- Positions the whole block across the terminal width per `banner.align`
  (`left`/`center`/`right`).
- Appends the enabled info lines (`show_user`, `show_host`, `show_os`,
  `show_datetime`, `show_uptime`, `show_shell`, `show_python`, `show_ip`).
- Prints to stdout via a Rich `Console`.

Both files are rendered from Jinja2 templates in `src/welchost/templates/` with
the config values baked in. **Users must never edit `welcome_banner.py`
directly** ‚Äî it is regenerated from `welchost.toml`.

## 11. DEV mode (sandbox)

Activation (all equivalent): `welchost --dev <cmd>`, `welchost <cmd> --dev`,
`WELCHOST_DEV=1 welchost`.

| Behavior | Normal | DEV |
|---|---|---|
| Config dir | `~/.config/ghostty/` | `./dev-home/.config/ghostty/` |
| `.zshrc` target | `~/.zshrc` | `./dev-home/.zshrc` |
| Ghostty check | required (warn if missing) | skipped (`ghostty_installed()` ‚Üí True) |
| TUI | Textual app | Textual app + (optional) textual console |
| File watcher | off | watchdog watches `src/welchost/` |
| Reset scope | real files | wipes `./dev-home/` only |

`dev-home/` is created automatically on first `--dev` run and is git-ignored.
Path resolution lives entirely in `detect.py`:

```python
# detect.py
DEV_MODE: bool = False
DEV_HOME = Path("dev-home")  # relative to cwd

def get_config_dir() -> Path:
    if DEV_MODE:
        return DEV_HOME / ".config" / "ghostty"
    xdg = os.environ.get("XDG_CONFIG_HOME")
    return (Path(xdg) / "ghostty") if xdg else Path.home() / ".config" / "ghostty"

def get_zshrc() -> Path:
    return (DEV_HOME / ".zshrc") if DEV_MODE else Path.home() / ".zshrc"

def ghostty_installed() -> bool:
    if DEV_MODE:
        return True
    return (Path("/Applications/Ghostty.app").exists()
            or Path("~/Applications/Ghostty.app").expanduser().exists()
            or shutil.which("ghostty") is not None)
```

Hot reload (`tui/dev_watcher.py`) is only imported/started when `DEV_MODE` is
True ‚Äî never in production.

## 12. What NOT to do (hard constraints)

- ‚ùå Never touch `~/.config/ghostty/config` or `config.ghostty`.
- ‚ùå Never call the system `figlet` binary ‚Äî always pyfiglet.
- ‚ùå Never blindly append to `~/.zshrc` ‚Äî check the sentinel first.
- ‚ùå Never assume `~/.config/ghostty/` exists ‚Äî always `mkdir -p`.
- ‚ùå Never let users edit `welcome_banner.py` directly ‚Äî it is generated.
- ‚ùå Never ship a prebuilt binary.
- ‚ùå Never import `welchost.tui` from a core module.
- ‚ùå Never hardcode `~`/paths ‚Äî use `detect.get_config_dir()` / `get_zshrc()`.
- ‚ùå Never use bare `print()` for user output ‚Äî use a Rich `Console`.

- **test_detect.py** ‚Äî Ghostty detection; DEV vs normal path resolution.
- **test_dev_mode.py** ‚Äî `--dev` writes to `dev-home/` not `~`; `--dev reset`
  wipes `dev-home/` only; `dev-home/` auto-created; `ghostty_installed()` True in DEV.
- **test_themes.py** ‚Äî all 12 themes load and every font name is a valid pyfiglet
  font.

Key invariants to protect: sentinel idempotency, DEV isolation, lossless config

## 14. Distribution & release

Published to three places on every release:
- **PyPI**: `pipx install welchost` / `pip install welchost` (primary).
- **Homebrew tap**: separate repo `scoobynko/homebrew-welchost`, formula at
  `Formula/welchost.rb` (`Language::Python::Virtualenv` +
  `virtualenv_install_with_resources`). Install:
  `brew install scoobynko/welchost/welchost`.

### Release automation (never run manually)

Conventional commits ‚Üí `python-semantic-release` detects the bump ‚Üí publishes.
- `fix:`/`chore:`/`perf:`/`refactor:` ‚Üí patch
- `feat:` ‚Üí minor
- `feat!:` / `BREAKING CHANGE:` ‚Üí major

Commit format: `type(scope): description`. Types: `feat fix chore docs test
refactor perf ci build`. Enforced by commitlint in CI + `conventional-pre-commit`
locally.

Workflows (`.github/workflows/`):
- **ci.yml** ‚Äî lint + ruff format check + pytest matrix (3.11/3.12/3.13) + commit lint on PRs.
- **release.yml** ‚Äî on push to main: test ‚Üí semantic-release ‚Üí PyPI publish ‚Üí
  bump Homebrew formula.
- **pr-check.yml** ‚Äî commit-message + PR-title conventional-commit validation.
- **homebrew-test.yml** ‚Äî on release: `brew install` smoke test.

Required GitHub secrets: `PYPI_TOKEN`, `HOMEBREW_TAP_TOKEN` (PAT, repo+workflow
on the tap repo), `WELCHOST_POSTHOG_KEY` (PostHog project key, baked into the
build at release time ‚Äî release.yml skips the bake step when unset, so telemetry
just stays off). `GITHUB_TOKEN` is automatic. `main` is branch-protected
(require PR + CI checks, no bypass).
